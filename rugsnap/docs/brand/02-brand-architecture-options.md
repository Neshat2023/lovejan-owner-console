# 02 — Brand Architecture Options

**Date:** 2026-07-23 · **Status:** Analysis + recommendation. Challenges the Bible v1.0 §2 choice on purpose (red-team mandate).

## The four models

### Option A — RugSnap is the company and the everything-brand
Stretch RugSnap across watches, jewelry, coins, art.
- **How categories arrive:** tabs inside RugSnap.
- **Equity:** one pool, but anchored to "rug."
- **SEO:** strong for rugs forever; incoherent for "rugsnap watch appraisal."
- **App Store:** one app (good).
- **Trademark:** carries the "Snap" adjacency question everywhere it grows (UNVERIFIED risk, counsel needed).
- **5-yr advantage:** zero migration cost. **5-yr risk:** the name is the ceiling; a watch collector bounces off "RugSnap" before the product can speak.
- **Verdict:** fine for 12–18 months; wrong endgame.

### Option B — House of brands: RugSnap, ArtSnap, CoinSnap, WatchSnap… (Bible v1.0's pick)
- **How categories arrive:** each vertical gets its own brand/app/landing on the shared platform.
- **Equity:** fragmented across N small brands; none compounds.
- **SEO/ASO:** every launch starts from domain authority zero and app-rank zero; content and links never pool.
- **App Store:** N apps to maintain, review, and rank — for a solo operator this is the killer.
- **Trademark:** multiplies the "Snap" exposure — a family of Snap-suffixed camera-adjacent apps is *more* likely to draw attention from Snap Inc. than one niche app (preliminary reasoning, UNVERIFIED; counsel).
- **Marketplace mechanics:** cross-category buyers (estate flows contain rugs *and* art *and* jewelry) get split across apps — directly damages our best journey (J4).
- **5-yr advantage:** per-niche resonance. **5-yr risk:** brand sprawl, diluted equity, multiplied TM risk, broken estate UX.
- **Verdict:** **red-team overturns Bible v1.0 here.** The Snap-family plan looks tidy on a slide and is operationally and legally the weakest option. The Bible's underlying *engineering* rule (category = configuration, no "rug" in code) stands; the multi-brand skin plan should not.

### Option C — One universal brand; rugs become a category inside it
- **How categories arrive:** categories inside one platform/app; "Rugs" is the flagship room.
- **Equity/SEO/ASO:** everything compounds in one place; one app.
- **Cost:** a new name must be created, cleared, and acquired; RugSnap's (small) equity must be bridged; rebrand execution risk.
- **5-yr advantage:** the only structure that matches the estate journey, the passport network, and a fundraise narrative. **5-yr risk:** naming/clearing is hard; a weak universal name is worse than a strong niche one.
- **Precedents (model knowledge):** eBay (collectibles→everything, one brand), StockX (sneakers→cards/watches, one brand), Chairish, TheRealReal — generalizing marketplaces overwhelmingly stay single-brand; durable multi-brand marketplace families are usually M&A artifacts (Match Group), not organic strategies.

### Option D — Endorsed hybrid: parent brand + "RugSnap by <Parent>"
- RugSnap keeps its niche pull and SEO; the parent accumulates platform equity; new categories launch as parent-branded categories (not new apps); RugSnap gradually becomes the rugs door into the parent platform.
- **5-yr advantage:** no equity is thrown away; migration is gradual and reversible at each step. **5-yr risk:** dual-brand maintenance for a while; endorsement lockups must be designed once, well.

## Recommendation (decisive)

**Adopt Option D now, converging to C at scale.** Concretely:
1. Select and secure a universal parent name via docs 03–05 (verification-gated).
2. RugSnap stays the consumer wedge exactly as-is; packaging becomes "RugSnap — by <Parent>" only after the parent name clears counsel.
3. All *new* categories launch inside the parent platform/app as categories — **never** as new Snap-branded apps.
4. Bible §2 is amended accordingly in v1.1 (this supersedes the v1.0 "house of brands" lean).
5. Trigger unchanged: parent selection completes before vertical #2 launch or first institutional raise, whichever first.

## Niche vs. general (mandated question, direct answer)

**Start niche (rugs), build general foundations** — option 2 of the mandate's three. Full-breadth launch fails on AI accuracy (per-category evals don't exist yet), liquidity (thin everywhere), and authentication ops (watch/jewelry fraud tiers unmet). Rugs-only-forever fails on TAM and on the estate journey (estates are inherently multi-category). The wedge-with-neutral-foundation path is the only one consistent with the evidence and the unit-economics caution in the v1.1 plan. This is what the codebase already implements (category is data), so the recommendation costs nothing to keep.
